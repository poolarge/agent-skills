# 计划与任务分解

## 概述

将工作分解为小的、可验证的任务，带有明确的验收标准。好的任务分解是可靠完成工作的 agent 和制造混乱的 agent 之间的区别。每个任务应足够小，能在一个专注的会话中实现、测试和验证。

## 何时使用

- 你有规格并需要将其分解为可实现单元
- 任务感觉太大或太模糊无法开始
- 工作需要跨多个 agent 或会话并行
- 需要向人类传达范围
- 实现顺序不明显

**何时不使用：** 范围明确的单文件变更，或规格已包含定义良好的任务。

## 计划流程

### 步骤 1：进入计划模式

在编写任何代码之前，以只读模式运作：

- 阅读规格和相关代码库部分
- 识别现有模式和约定
- 映射组件之间的依赖关系
- 记录风险和未知因素

**计划期间不要编写代码。** 产出是计划文档，不是实现。

### 步骤 2：识别依赖图

映射什么依赖什么：

```
数据库 schema
    │
    ├── API 模型/类型
    │       │
    │       ├── API 端点
    │       │       │
    │       │       └── 前端 API 客户端
    │       │               │
    │       │               └── UI 组件
    │       │
    │       └── 验证逻辑
    │
    └── 种子数据 / 迁移
```

实现顺序遵循依赖图自底向上：先构建基础。

### 步骤 3：垂直切分

不要先构建所有数据库，再构建所有 API，再构建所有 UI——而是一次构建一条完整的功能路径：

**差的（水平切分）：**
```
任务 1：构建整个数据库 schema
任务 2：构建所有 API 端点
任务 3：构建所有 UI 组件
任务 4：连接所有内容
```

**好的（垂直切分）：**
```
任务 1：用户可以创建账户（schema + API + 注册 UI）
任务 2：用户可以登录（认证 schema + API + 登录 UI）
任务 3：用户可以创建任务（任务 schema + API + 创建 UI）
任务 4：用户可以查看任务列表（查询 + API + 列表视图 UI）
```

每个垂直切片交付可工作、可测试的功能。

### 步骤 4：编写任务

每个任务遵循此结构：

```markdown
## 任务 [N]：[简短描述性标题]

**描述：** 一段话说明此任务完成什么。

**验收标准：**
- [ ] [具体、可测试的条件]
- [ ] [具体、可测试的条件]

**验证：**
- [ ] 测试通过：`npm test -- --grep "feature-name"`
- [ ] 构建成功：`npm run build`
- [ ] 手动检查：[需要验证什么]

**依赖：** [依赖的任务编号，或"无"]

**可能涉及的文件：**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**预估范围：** [小：1-2 个文件 | 中：3-5 个文件 | 大：5+ 个文件]
```

### 步骤 5：排序与检查点

安排任务使得：

1. 依赖关系得到满足（先构建基础）
2. 每个任务让系统保持在可工作状态
3. 每 2-3 个任务后有验证检查点
4. 高风险任务排在前面（快速失败）

添加明确的检查点：

```markdown
## 检查点：任务 1-3 完成后
- [ ] 所有测试通过
- [ ] 应用构建无错误
- [ ] 核心用户流程端到端可工作
- [ ] 与人类评审后再继续
```

## 任务大小指南

| 大小 | 文件数 | 范围 | 示例 |
|------|--------|------|------|
| **XS** | 1 | 单个函数或配置变更 | 添加验证规则 |
| **S** | 1-2 | 一个组件或端点 | 添加新 API 端点 |
| **M** | 3-5 | 一个功能切片 | 用户注册流程 |
| **L** | 5-8 | 多组件功能 | 带筛选和分页的搜索 |
| **XL** | 8+ | **太大——需进一步拆分** | — |

如果任务为 L 或更大，应拆分为更小的任务。agent 在 S 和 M 任务上表现最佳。

**何时进一步拆分任务：**
- 需要超过一个专注会话（约 2+ 小时的 agent 工作）
- 无法用 3 个或更少的要点描述验收标准
- 涉及两个或更多独立子系统（例如，认证和计费）
- 发现自己在任务标题中写了"和"（这是两个任务的信号）

## 计划文档模板

```markdown
# 实现计划：[功能/项目名称]

## 概述
[一段话总结我们正在构建什么]

## 架构决策
- [关键决策 1 及理由]
- [关键决策 2 及理由]

## 任务列表

### 阶段 1：基础
- [ ] 任务 1: ...
- [ ] 任务 2: ...

### 检查点：基础
- [ ] 测试通过，构建干净

### 阶段 2：核心功能
- [ ] 任务 3: ...
- [ ] 任务 4: ...

### 检查点：核心功能
- [ ] 端到端流程可工作

### 阶段 3：完善
- [ ] 任务 5: ...
- [ ] 任务 6: ...

### 检查点：完成
- [ ] 所有验收标准满足
- [ ] 准备评审

## 风险与缓解
| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| [风险] | [高/中/低] | [策略] |

## 待解决问题
- [需要人类输入的问题]
```

## 并行化机会

当多个 agent 或会话可用时：

- **可安全并行：** 独立功能切片、已实现功能的测试、文档
- **必须顺序执行：** 数据库迁移、共享状态变更、依赖链
- **需要协调：** 共享 API 契约的功能（先定义契约，再并行）

## 常见合理化说辞

| 合理化说辞 | 现实 |
|---|---|
| "我边做边想" | 这就是导致混乱和返工的方式。10 分钟的计划节省数小时。 |
| "任务很明显" | 仍然要写下来。明确的任务能暴露隐藏的依赖和被遗忘的边界情况。 |
| "计划是额外开销" | 计划就是任务本身。没有计划的实现只是在打字。 |
| "我都能记在脑子里" | 上下文窗口是有限的。书面计划能跨越会话边界和压缩而存续。 |

## 危险信号

- 没有书面任务列表就开始实现
- 任务说"实现功能"但没有验收标准
- 计划中没有验证步骤
- 所有任务都是 XL 大小
- 任务之间没有检查点
- 未考虑依赖顺序

## 验证

在开始实现之前，确认：

- [ ] 每个任务都有验收标准
- [ ] 每个任务都有验证步骤
- [ ] 任务依赖已识别且顺序正确
- [ ] 没有任务涉及超过约 5 个文件
- [ ] 主要阶段之间存在检查点
- [ ] 人类已审查并批准计划
