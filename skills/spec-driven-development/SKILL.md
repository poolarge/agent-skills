# 规格驱动开发

## 概述

在编写任何代码之前，先编写结构化的规格说明。规格说明是你和人类工程师之间的共同事实来源——它定义了我们要构建什么、为什么构建，以及如何确认完成。没有规格的代码就是猜测。

## 何时使用

- 开始新项目或新功能
- 需求模糊或不完整
- 变更涉及多个文件或模块
- 即将做出架构决策
- 实现任务预计超过 30 分钟

**何时不使用：** 单行修复、拼写纠正，或需求明确且自包含的变更。

## 门控工作流

规格驱动开发有四个阶段。在当前阶段验证通过之前，不要推进到下一阶段。

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
   │          │        │          │
   ▼          ▼        ▼          ▼
 人类       人类     人类       人类
 评审       评审     评审       评审
```

### 阶段 1：规格定义

从高层愿景开始。向人类提出澄清问题，直到需求变得具体。

**立即暴露假设。** 在编写任何规格内容之前，列出你正在做的假设：

```
我正在做的假设：
1. 这是一个 Web 应用（不是原生移动应用）
2. 认证使用基于会话的 cookie（不是 JWT）
3. 数据库是 PostgreSQL（基于现有 Prisma schema）
4. 我们只针对现代浏览器（不支持 IE11）
→ 现在纠正我，否则我将按这些假设继续。
```

不要悄悄填补模糊的需求。规格的全部目的就是在代码编写*之前*暴露误解——假设是最危险的误解形式。

**编写涵盖以下六个核心领域的规格文档：**

1. **目标** — 我们在构建什么，为什么？用户是谁？成功的定义是什么？

2. **命令** — 带有标志的完整可执行命令，而不仅是工具名称。
   ```
   构建: npm run build
   测试: npm test -- --coverage
   代码检查: npm run lint --fix
   开发: npm run dev
   ```

3. **项目结构** — 源代码在哪里，测试在哪里，文档在哪里。
   ```
   src/           → 应用源代码
   src/components → React 组件
   src/lib        → 共享工具函数
   tests/         → 单元和集成测试
   e2e/           → 端到端测试
   docs/          → 文档
   ```

4. **代码风格** — 一个展示你风格的真实代码片段胜过三段描述。包括命名约定、格式化规则和良好输出的示例。

5. **测试策略** — 使用什么框架，测试放在哪里，覆盖率期望，不同测试级别对应不同关注点。

6. **边界** — 三层体系：
   - **始终做：** 提交前运行测试，遵循命名约定，验证输入
   - **先问再做：** 数据库 schema 变更，添加依赖，更改 CI 配置
   **绝不做：** 提交密钥，编辑 vendor 目录，未经批准删除失败的测试

**规格模板：**

```markdown
# 规格：[项目/功能名称]

## 目标
[构建什么以及为什么。用户故事或验收标准。]

## 技术栈
[框架、语言、关键依赖及其版本]

## 命令
[构建、测试、代码检查、开发 — 完整命令]

## 项目结构
[目录布局及说明]

## 代码风格
[示例代码片段 + 关键约定]

## 测试策略
[框架、测试位置、覆盖率要求、测试级别]

## 边界
- 始终做: [...]
- 先问再做: [...]
- 绝不做: [...]

## 成功标准
[如何确认完成 — 具体、可测试的条件]

## 待解决问题
[需要人类输入的未解决问题]
```

**将指令重构为成功标准。** 当收到模糊需求时，将其转化为具体条件：

```
需求："让仪表盘更快"

重构为成功标准：
- 仪表盘 LCP < 2.5s（4G 连接）
- 初始数据加载在 < 500ms 内完成
- 加载期间无布局偏移（CLS < 0.1）
→ 这些是正确的目标吗？
```

这让你能够循环、重试和解决明确目标，而不是猜测"更快"的含义。

### 阶段 2：计划

基于验证过的规格，生成技术实现计划：

1. 识别主要组件及其依赖关系
2. 确定实现顺序（必须先构建什么）
3. 记录风险和缓解策略
4. 识别哪些可以并行构建，哪些必须顺序执行
5. 在各阶段之间定义验证检查点

计划应该可审查：人类应该能够阅读并说"是的，这是正确的方法"或"不，修改 X"。

### 阶段 3：任务

将计划分解为离散的、可实现的任务：

- 每个任务应在一个专注的会话中可完成
- 每个任务有明确的验收标准
- 每个任务包含验证步骤（测试、构建、手动检查）
- 任务按依赖关系排序，而非按感知的重要性排序
- 任何任务不应涉及修改超过约 5 个文件

**任务模板：**
```markdown
- [ ] 任务：[描述]
  - 验收：[完成时必须满足的条件]
  - 验证：[如何确认 — 测试命令、构建、手动检查]
  - 文件：[将涉及的文件]
```

### 阶段 4：实现

按照 `incremental-implementation` 和 `test-driven-development` 技能逐个执行任务。使用 `context-engineering` 在每个步骤加载正确的规格部分和源文件，而不是用整个规格淹没 agent。

## 保持规格活跃

规格是活文档，不是一次性产物：

- **决策变更时更新** — 如果你发现数据模型需要变更，先更新规格，再实现。
- **范围变更时更新** — 新增或削减的功能应反映在规格中。
- **提交规格** — 规格与代码一起纳入版本控制。
- **在 PR 中引用规格** — 链接回每个 PR 实现的规格部分。

## 常见合理化说辞

| 合理化说辞 | 现实 |
|---|---|
| "这很简单，不需要规格" | 简单任务不需要*长*规格，但仍需要验收标准。两行规格就够了。 |
| "我写完代码再写规格" | 那是文档，不是规格。规格的价值在于在代码之前强制明确。 |
| "规格会拖慢我们" | 15 分钟的规格避免了数小时的返工。15 分钟的瀑布流胜过 15 小时的调试。 |
| "需求反正会变" | 这正是规格是活文档的原因。过时的规格仍比没有规格好。 |
| "用户知道自己想要什么" | 即使明确的需求也有隐含假设。规格暴露这些假设。 |

## 危险信号

- 没有任何书面需求就开始写代码
- 在澄清"完成"的含义之前就问"我应该直接开始构建吗？"
- 实现规格或任务列表中未提及的功能
- 做出架构决策但没有记录
- 因为"显然要构建什么"而跳过规格

## 验证

在推进到实现之前，确认：

- [ ] 规格覆盖了所有六个核心领域
- [ ] 人类已审查并批准规格
- [ ] 成功标准具体且可测试
- [ ] 边界（始终做/先问再做/绝不做）已定义
- [ ] 规格已保存到仓库中的文件
